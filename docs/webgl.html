BadWolf + Emscripten

GPU-Accelerated Terminal UI

When compiled to WebAssembly with Emscripten, BadWolf apps can use WebGL for hardware-accelerated braille canvas rendering — smooth 60 fps animations with GPU-drawn glyph sprites.

Usage

WebGLScope — RAII Integration

WebGLScope automatically initialises WebGL on entry and shuts it down on exit. On native builds it is a harmless no-op.

// Works on native AND in the browser — #ifdef handled internally #include "ftxui/ui/webgl_renderer.hpp" #include "ftxui/ui/wasm_bridge.hpp" int main() { // RAII: Init on construction, Shutdown on destruction ftxui::ui::WebGLScope gl("badwolf-canvas"); if (gl.ok) { // GPU path — only reaches here in WASM with WebGL support ftxui::ui::WebGLRenderer::Clear(0, 0, 0); ftxui::ui::WebGLRenderer::RenderBrailleCanvas(dots, width, height); ftxui::ui::WebGLRenderer::Present(); } else { // CPU fallback — terminal rendering as usual } // RunApp: native = RunFullscreen, WASM = emscripten_set_main_loop ftxui::ui::RunApp([&] { return myComponent; }); }
How It Works

Braille Dots as GPU Sprites

Batch Rendering

All set braille dots in a frame are gathered into a single VBO of quad vertices. One glDrawArrays(GL_TRIANGLES, …) call renders every dot — GPU throughput scales to millions of dots/frame.

Zero Native Overhead

The #ifdef __EMSCRIPTEN__ guard means every WebGL/Emscripten header is excluded from native compilations. The stub functions are inlined to nothing by the optimiser.

Device Pixel Ratio

The HTML shell sets canvas.width = clientWidth × devicePixelRatio so braille dots are rendered at native resolution on HiDPI/Retina displays without blurring.

Dual Canvas

Two stacked <canvas> elements: one carries the FTXUI BadWolf terminal text output, the other the WebGL overlay. Transparency lets both layers composit naturally.
