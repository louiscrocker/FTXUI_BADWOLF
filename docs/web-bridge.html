HTTP · WebSocket · xterm.js · Zero Client Code

Run in the Browser

Serve any BadWolf TUI app in a browser over HTTP + WebSocket — zero client code needed. The server renders the FTXUI component to ANSI escape sequences and streams live updates to xterm.js. Keyboard input flows back to the app.

Architecture

BadWolf App ──renders──▶ ftxui::Screen ──ANSI──▶ WebSocketServer ──▶ Browser (xterm.js)
Browser keyboard events ──▶ WebSocket ──▶ ftxui::Event ──▶ App

  ┌─────────────────────────────────────────────────────────────────┐
  │  WebBridge                                                      │
  │   ├─ HTTP server  →  GET /   serves embedded xterm.js page      │
  │   │                  GET /ws upgrades to WebSocket (RFC 6455)   │
  │   ├─ Render loop  →  component→Render() @ N fps                 │
  │   │                  Screen::ToString() → ANSI string           │
  │   │                  broadcast to all WS clients                │
  │   └─ Event loop   →  WS frames → ftxui::Event → component       │
  └─────────────────────────────────────────────────────────────────┘

Quick Start

One-liner integration

Replace your int main() with RunWebOrTerminal — no other changes required. The app runs normally in the terminal; pass --web to serve it in a browser instead. 

#include "ftxui/ui/web_bridge.hpp"

int main(int argc, char** argv) {
  ftxui::ui::RunWebOrTerminal(argc, argv, [] {
    // Return your existing root component unchanged.
    return MyComponent();
  });
}

Run without --web: normal terminal app.
Run with --web: serves at http://localhost:7681.

CLI — badwolf serve

badwolf serve myapp.cpp

Compiles the file with web bridge support and starts the HTTP server. Open the displayed URL in your browser.

API Reference

WebBridgeConfig

Field Default Description
port 7681 HTTP + WebSocket port. Auto-increments up to +10 on conflict.
host "0.0.0.0" Bind address.
cols 220 Terminal width served to the browser.
rows 50 Terminal height served to the browser.
fps 30 Maximum render rate (frames per second).
open_browser false Automatically open the browser on start.
title "BadWolf" HTML page <title>.

WebBridge class

class WebBridge {
  // Blocks — serves until Ctrl+C or Stop().
  void Run(ftxui::Component root);

  // Non-blocking: start server threads, return immediately.
  void Start(ftxui::Component root);
  void Stop();
  bool IsRunning() const;

  int         port() const;   // actual bound port
  std::string url()  const;   // "http://localhost:7681"

  // Inject an ftxui::Event (useful for testing).
  void InjectEvent(ftxui::Event e);
};

RunWebOrTerminal helper

void RunWebOrTerminal(
    int argc, char** argv,
    std::function<ftxui::Component()> factory,
    WebBridgeConfig cfg = {});

If --web is present in argv, starts the WebBridge and blocks. Otherwise calls RunFullscreen(factory()) for normal terminal mode.

Technical Notes

🔒 POSIX sockets only

No external HTTP or WebSocket libraries. Uses raw POSIX socket(), bind(), accept(). The WebSocket handshake implements SHA-1 (RFC 3174) and Base64 from scratch.

🔄 Port conflict recovery

If the configured port is in use, the server automatically tries port+1 through port+10 before giving up.

📺 xterm.js from CDN

The embedded HTML page loads xterm.js and xterm-addon-fit from jsDelivr CDN — client-side only. The server itself has no JS dependency.

⌨️ Key translation

xterm.js key sequences are translated to ftxui::Event: arrow keys, Home/End, Page Up/Down, Insert/Delete, Tab, Backspace, Escape, Return, and arbitrary UTF-8 characters.

🧵 Thread model

One render thread drives the component at the configured FPS. One server/accept thread. Each WebSocket client gets its own detached thread for I/O.

🎯 Zero app changes

The component is rendered using ftxui::Render() directly — no terminal hooks, no alt-screen. Events are injected via component->OnEvent().
