Under the Hood

How FTXUI achieves sub-millisecond renders, 3D projections, adaptive resolution, and thread-safe reactivity — all in pure C++20.

Section 01

Architecture Overview

FTXUI is organized in four layers. Each layer builds on the one below. The ftxui::ui layer is additive — it never modifies the layers beneath it.

ftxui::ui    (BadWolf Layer — high-level API)
Text & Richness: TypewriterText │ RichText │ AnsiParser  │  Data: JsonValue │ ReactiveJson │ JsonPath  │  AI: NLParser │ UIGenerator │ LLMAdapter
Reactive: Reactive<T> │ Bind<T> │ ReactiveList<T>  │  Visualization: BrailleCanvas │ LineChart │ GeoMap │ GlobeMap │ GalaxyMap  │  Net: CollabServer │ LiveSource<T>
Themes: LCARS │ Imperial │ Enterprise │ Rebel │ Matrix │ Dracula  │  Animation: AnimationLoop │ Tween │ ParticleSystem  │  WASM: WebGLRenderer │ WasmBridge
▼
ftxui::component    (interactive event handling)
Button │ Input │ Renderer │ Container │ CatchEvent │ Modal
▼
ftxui::dom    (layout and styling elements)
hbox │ vbox │ text │ canvas │ border │ color │ size │ flex │ separator
▼
ftxui::screen    (terminal I/O, pixels, events)
Terminal I/O │ Color │ Pixel │ Event │ Screen │ Canvas (braille)
Design principle: Each layer is independently usable. You can use just ftxui::dom to render static layouts, or just ftxui::screen for raw terminal access. The ftxui::ui layer adds the high-level builder API on top.

Single-Header (BADWOLF_IMPLEMENTATION)

Drop dist/ftxui.hpp into your project. No cmake, no linking, no build system — just one file and one define. 

// In exactly ONE .cpp file:
#define BADWOLF_IMPLEMENTATION
#include "ftxui.hpp"

// In all other files that need declarations:
#include "ftxui.hpp"

// Compile with no extra link step:
// g++ -std=c++20 -pthread -o myapp myapp.cpp

This is the STB-style single-header pattern. The header contains all declarations unconditionally, and all .cpp implementations wrapped in #ifdef BADWOLF_IMPLEMENTATION.

Section 02

Adaptive Braille Canvas Engine

The Braille Resolution Trick

Unicode braille characters encode 2×4 dots per character cell. This gives FTXUI 8× the resolution of normal character rendering — without any special terminal support. 

// A standard 80×24 terminal in braille resolution:
// Characters: 80 × 24 = 1,920 cells
// Braille dots: (80×2) × (24×4) = 160 × 96 = 15,360 addressable dots

// Each braille character encodes an 8-bit pattern:
// ⡿ = 0b11111110 — 7 of 8 dots set
// ⢿ = 0b11111101 — 7 of 8 dots set, different column
// ⣿ = 0b11111111 — all 8 dots set (solid block)

// Mapping: unicode codepoint = U+2800 + bitmask
// Bit layout per cell:
//   col0 col1
//   bit0 bit3  (row 0)
//   bit1 bit4  (row 1)
//   bit2 bit5  (row 2)
//   bit6 bit7  (row 3)

Adaptive Sizing with canvas(w, h, fn) | flex

The key insight: canvas(w, h, fn) specifies minimum dimensions. The CanvasFunctionNode::Render() creates a Canvas at the actual box size allocated by the layout engine. Your draw function reads c.width() and c.height() to draw at full resolution. 

// This chart fills ALL available terminal space at MAX braille resolution:
auto element = canvas(2, 4, [&](Canvas& c) {
    int w = c.width();   // actual dot-width (always even)
    int h = c.height();  // actual dot-height (always mult of 4)

    // Scale data to actual canvas size:
    for (int i = 0; i < (int)data.size() - 1; i++) {
        int x1 = i * w / data.size();
        int y1 = h - (int)(data[i] * h / max_val);
        int x2 = (i+1) * w / data.size();
        int y2 = h - (int)(data[i+1] * h / max_val);
        c.DrawPointLine(x1, y1, x2, y2, Color::Red());
    }
}) | flex;  // <-- this makes it fill all space

Section 03

GeoJSON Rendering Pipeline

Step-by-Step Pipeline

1
Parse JSON → Feature Vector
Iterate the GeoJSON features array. For each feature, extract geometry.type (Point, LineString, Polygon, MultiPolygon) and properties (name, etc.). Store as vector<Feature>.
2
Project lon/lat → Canvas Pixels
Equirectangular projection scaled to canvas dots.
// lon ∈ [-180, 180] → px ∈ [0, dot_w]
int px = (lon + 180.0) / 360.0 * dot_w;
// lat ∈ [-90, 90]   → py ∈ [0, dot_h]
int py = (90.0 - lat) / 180.0 * dot_h;
3
Anti-Wrap Guard
Countries that cross the ±180° meridian (Russia, USA Alaska) would draw a line across the entire map. The anti-wrap guard detects and skips these segments.
if (std::abs(px - prev_px) > dot_w / 2) {
    prev_px = px; prev_py = py;
    continue; // skip anti-meridian segment
}
4
Overlay Rendering
After the base map is drawn, overlays are rendered on top: marker pins at specific lon/lat, text labels, and animated arcs between cities.
5
Great-Circle Arc Rendering
Arcs between cities use spherical linear interpolation (slerp) on the unit sphere, then project each interpolated point to canvas coordinates. Creates the characteristic curved flight path appearance.
// Slerp 50 points along the great circle
for (int i = 0; i < 50; i++) {
    double t = i / 49.0;
    auto [lon, lat] = slerp_geodesic(p1, p2, t);
    int px = (lon + 180.0) / 360.0 * dot_w;
    int py = (90.0 - lat) / 180.0 * dot_h;
    // draw point...
}
Section 04

Globe Orthographic Projection

The 3D spinning globe uses classical orthographic projection with a visibility test. All math is in ~50 lines of C++.

The Math

// Given: center (λ₀, φ₀) — current viewing longitude/latitude
// Given: point  (λ, φ)   — the point to project

// Step 1: Visibility test — is this point on the visible hemisphere?
double cos_c = sin(φ₀) * sin(φ)
              + cos(φ₀) * cos(φ) * cos(λ - λ₀);

if (cos_c <= 0.0) continue; // point is on the back hemisphere — skip

// Step 2: Project to 2D plane
double x = cos(φ) * sin(λ - λ₀);
double y = cos(φ₀) * sin(φ)
         - sin(φ₀) * cos(φ) * cos(λ - λ₀);

// Step 3: Scale to canvas pixels
double radius = std::min(dot_w, dot_h) / 2.0 * 0.9;
int px = dot_w / 2 + x * radius;
int py = dot_h / 2 - y * radius;  // y-flip for screen coords

c.DrawPoint(px, py, theme_color);

Animation Architecture

The globe spins via a dedicated background thread, not by blocking the event loop.

// Background thread updates rotation center and posts a refresh event
void GlobeAnimationThread(double deg_per_sec,
                           std::atomic<double>& lambda0,
                           std::atomic<bool>& running) {
    auto last = std::chrono::steady_clock::now();
    while (running.load()) {
        auto now = std::chrono::steady_clock::now();
        double dt = std::chrono::duration<double>(now - last).count();
        last = now;

        double new_lon = std::fmod(lambda0 + deg_per_sec * dt, 360.0);
        lambda0.store(new_lon);

        // Trigger UI redraw (zero-copy event)
        App::Active()->Post({});

        // ~20fps
        std::this_thread::sleep_for(std::chrono::milliseconds(50));
    }
}
Section 05

Reactive State System

Thread-safe, re-entrant, zero-allocation on the hot path.

Reactive<T> Implementation

template<typename T>
class Reactive {
    T value_;
    std::mutex mutex_;
    std::map<int, std::function<void(T)>> listeners_;
    int next_id_ = 0;

public:
    void Set(T new_value) {
        // Copy value and listener map under lock
        decltype(listeners_) snapshot;
        {
            std::lock_guard<std::mutex> lock(mutex_);
            value_ = std::move(new_value);
            snapshot = listeners_;  // copy before calling
        }

        // Call listeners WITHOUT holding the lock
        // — this is what makes it re-entrant safe
        for (auto& [id, fn] : snapshot) {
            fn(value_);
        }

        // Post a zero-copy event to trigger UI re-render
        if (auto app = App::TryActive()) {
            app->Post(Event::Custom);
        }
    }
};

ReactiveGroup — Batch Updates

Uses a thread-local batch depth counter to suppress per-mutation refreshes and coalesce them into a single redraw.

thread_local int batch_depth = 0;

void ReactiveGroup::Batch(std::function<void()> fn) {
    batch_depth++;
    fn();         // all Set() calls inside suppress refresh
    batch_depth--;
    if (batch_depth == 0) {
        // Single refresh for all mutations
        App::Active()->Post(Event::Custom);
    }
}

// Inside Reactive::Set():
if (batch_depth == 0) {
    app->Post(Event::Custom); // only if not in batch
}
Section 06

Virtual Rendering — O(visible)

50,000 rows render as fast as 50. The trick: use reflect(box) to find out how many rows fit in the current terminal, then only render those.

Element VirtualListNode::Render() {
    // On first render, box_ is empty. Render placeholder.
    if (!box_.y_max) return placeholder_element();

    // From the previous frame: how many rows fit?
    int visible_rows = box_.y_max - box_.y_min;
    int end = std::min(scroll_offset_ + visible_rows,
                      (int)items_.size());

    // Only render items[scroll_offset_..end)
    Elements rendered;
    rendered.reserve(visible_rows);
    for (int i = scroll_offset_; i < end; i++) {
        bool selected = (i == selected_index_);
        rendered.push_back(renderer_(items_[i], selected));
    }

    return vbox(std::move(rendered)) | reflect(box_);
    //                                    ^^^ captures box next frame
}

// Result: render time is O(visible_rows), not O(total_rows)
// 50,000 rows → same as 50 visible rows.

C API Design & FFI

Language-agnostic ABI via opaque handles. Any language with C FFI (Rust, Python, Go, Zig, D, ...) can use FTXUI.

Opaque Handle Types

// ftxui-c.h — the C API surface
#ifdef __cplusplus
extern "C" {
#endif

// Opaque pointer types
typedef struct ftxui_element_s*   ftxui_element_t;
typedef struct ftxui_component_s* ftxui_component_t;

// Internally: malloc'd struct containing shared_ptr
struct ftxui_element_s {
    ftxui::Element value;  // = shared_ptr<Node>
};

// Factory functions
ftxui_element_t   ftxui_text(const char* text);
ftxui_element_t   ftxui_hbox(ftxui_element_t* elems, int count);
ftxui_component_t ftxui_button(const char* label, void (*cb)(void*), void* userdata);
void              ftxui_run_fullscreen(ftxui_component_t root);

// Caller MUST free returned handles
void ftxui_free_element(ftxui_element_t);
void ftxui_free_component(ftxui_component_t);

#ifdef __cplusplus
}
#endif

Python Binding — GIL Safety

The Python bindings release the GIL during run_fullscreen so other Python threads can run. The FTXUI event callbacks re-acquire it.

// In the pybind11 module
m.def("run_fullscreen", [](py::object component) {
    // Convert Python component to C++ Component
    auto cpp_comp = python_to_component(component);

    {
        // Release GIL: FTXUI's event loop runs in C++
        py::gil_scoped_release release;
        ftxui::ui::RunFullscreen(cpp_comp);
    } // GIL re-acquired when RunFullscreen returns
}, py::arg("component"));

// Callbacks acquire GIL before calling Python:
auto wrap_callback = [](py::object fn) {
    return [fn]() {
        py::gil_scoped_acquire acquire;
        fn();
    };
};
